1. Stockage dans OpenStack

Trois modèles de stockage

Object, block, fichier : on choisit selon l'usage, pas l'inverse

Object

Swift

Stocke des objets (fichiers + métadonnées) accessibles via HTTP REST. Pas de système de fichiers, pas de hiérarchie réelle, scalable à l'infini.

  • Backups
  • Images Glance
  • Archives, médias
  • Logs applicatifs
  • Données web statiques

API native : Swift API + S3 compatible

Block

Cinder

Provisionne des volumes attachables à des VM Nova comme des disques durs. Très bonnes performances I/O, exclusif (un volume = une VM).

  • Disque système des VM
  • Bases de données
  • Volumes data persistants
  • Stockage Kubernetes (CSI)
  • Boot from volume

Backends : Ceph, LVM, NetApp, Pure, EMC…

Fichier

Manila

Fournit des partages de fichiers (NFS, CIFS, CephFS) accessibles depuis plusieurs VM simultanément.

  • Home dirs partagés
  • Sauvegardes mutualisées
  • Échange entre microservices
  • Lift-and-shift apps legacy

Backends : CephFS, NetApp, GlusterFS

💡 Ce module couvre Swift et Cinder, les deux briques de stockage les plus utilisées. Manila est abordée brièvement en fin de module.

2. Swift : architecture

Le Ring : cœur de Swift

🎯 Rôle du Ring

  • Mapping déterministe : pour un objet donné, dire sur quels disques il vit
  • Hashing cohérent : md5(path) → partition → liste de disques
  • Trois rings indépendants : account, container, object
  • Distribué à tous les nœuds (proxy + storage), identique partout
  • Modifié uniquement via swift-ring-builder

🧩 Concepts du Ring

  • Partition : unité logique (typiquement 2¹⁸ = 262 144 partitions)
  • Replica : copie d'une partition (3 par défaut)
  • Zone : groupe de disques tolérant pannes (rack, salle…)
  • Region : groupe de zones (datacenter)
  • Device : un disque physique sur un node
  • Weight : poids relatif d'un device dans la distribution

2. Swift : architecture

Réplication et eventual consistency

🔄 Processus de réplication

  • replicator compare en continu les hashes des partitions entre nodes (rsync ou ssync)
  • Si un disque tombe, ses partitions sont reconstruites ailleurs depuis les autres répliques
  • auditor détecte la corruption silencieuse (bit-rot) et marque le fichier "quarantined"
  • updater rejoue les opérations en échec (mise à jour container/account différée)
  • reaper nettoie les accounts supprimés

⚖️ Quorum

Avec R=3 répliques, le quorum d'écriture est W=2. La lecture peut tomber sur une partition pas encore synchronisée — Swift fait alors fallback sur une autre réplique.

⚠️ Eventual consistency

Après un PUT, une lecture immédiate peut renvoyer la version précédente sur quelques secondes. Pour les usages où ce comportement n'est pas acceptable (cache code, config-runtime), Cinder est un meilleur choix.

🧮 Erasure Coding (alternative)

  • À la place de N répliques complètes, on découpe l'objet en k+m fragments
  • k fragments = données, m = parité (Reed-Solomon)
  • Exemple : EC 8+4 tolère 4 pertes pour 1.5× la taille (vs 3× en réplication)
  • Économie d'espace, latence accrue, idéal pour objets > 1 Mo
  • Configuré par storage policy au niveau du container

Bonne pratique : 3 storage nodes minimum (1 par zone), chacun avec plusieurs disques dédiés (XFS) montés sous /srv/node/<dev>.

💾 Préparation des disques sur les storage

# Sur chaque storage-XX

# 1. Identifier les disques
lsblk -d -o NAME,SIZE,TYPE
# sdb 4T disk
# sdc 4T disk

# 2. Formater en XFS (recommandé Swift)
mkfs.xfs -L swift_d1 -i size=1024 -n size=8192 /dev/sdb
mkfs.xfs -L swift_d2 -i size=1024 -n size=8192 /dev/sdc

# 3. Créer les points de montage
mkdir -p /srv/node/d1 /srv/node/d2

# 4. fstab : noatime + nodiratime + nobarrier
cat >> /etc/fstab <<EOF
LABEL=swift_d1 /srv/node/d1 xfs noatime,nodiratime 0 2
LABEL=swift_d2 /srv/node/d2 xfs noatime,nodiratime 0 2
EOF

mount -a
chown -R kolla:kolla /srv/node

Préparer les rings avant le déploiement

Kolla génère les rings depuis un fichier de description des devices

📁 Structure attendue

# /etc/kolla/config/swift/
swift/
├── account.builder
├── container.builder
├── object.builder
│
├── account.ring.gz
├── container.ring.gz
├── object.ring.gz
│
└── swift.conf            # overrides middleware

Soit on fournit des builders pré-générés, soit on les construit avec un conteneur Kolla utilitaire.

⚙️ Paramètres du ring

  • part_power : 2^N partitions (typiquement 18 = 262 144)
  • replicas : 3 par défaut
  • min_part_hours : 1h avant qu'une partition puisse à nouveau bouger

🛠️ Génération avec swift-ring-builder

# Conteneur utilitaire fourni par Kolla
docker run --rm -it \
  -v /etc/kolla/config/swift:/etc/kolla/config/swift \
  kolla/swift-base:$openstack_release bash
# Dans le conteneur :
cd /etc/kolla/config/swift

# 1. Créer les builders (part_power 10 = lab)
swift-ring-builder account.builder   create 10 3 1
swift-ring-builder container.builder create 10 3 1
swift-ring-builder object.builder    create 10 3 1

# 2. Ajouter chaque device au ring
swift-ring-builder account.builder add \
  --region 1 --zone 1 --ip 10.10.0.31 \
  --port 6202 --device d1 --weight 100
# Idem pour container (port 6201) et object (6200)

# 3. Rebalance et générer les .ring.gz
swift-ring-builder account.builder rebalance
swift-ring-builder container.builder rebalance
swift-ring-builder object.builder rebalance

Pools de stockage : storage policies

Mélanger réplication classique et erasure coding dans le même cluster

🎯 Principe

  • Une storage policy = un ring object + ses paramètres (replication ou erasure coding)
  • Un container choisit sa policy à la création — immuable ensuite
  • Permet de faire cohabiter dans le même cluster : SSD vs HDD, réplication 3× vs EC 8+4, zones géographiques différentes
  • Optimise les coûts selon la valeur de la donnée

📝 swift.conf — extrait override

# /etc/kolla/config/swift/swift.conf
[storage-policy:0]
name = default-3replicas
default = yes
policy_type = replication

[storage-policy:1]
name = ssd-fast
policy_type = replication
deprecated = false

[storage-policy:2]
name = archive-ec
policy_type = erasure_coding
ec_type = liberasurecode_rs_vand
ec_num_data_fragments = 8
ec_num_parity_fragments = 4
ec_object_segment_size = 1048576

💻 Création de containers par policy

# Créer un container avec policy SSD
swift post hot-data \
  --header "X-Storage-Policy: ssd-fast"

# Vérifier la policy
swift stat hot-data
Account: AUTH_xxx
Container: hot-data
X-Storage-Policy: ssd-fast

# Container EC pour archives froides
swift post archives \
  --header "X-Storage-Policy: archive-ec"

# Migration entre policies =
# copier l'objet dans un nouveau container
# puis supprimer l'ancien
# (la policy est immuable)
swift copy --destination /archives/db.tgz \
  hot-data db.tgz
swift delete hot-data db.tgz

Chaque policy a son propre ring object — il faut en créer un par policy déclarée.

Gestion des pools : zones et regions

🌐 Stratégie

  • Zone = unité de panne (rack, salle)
  • Le ring garantit qu'aucune zone n'a deux répliques d'une même partition
  • Region = datacenter complet, réplication asynchrone
  • Permet du Disaster Recovery géographique

⚖️ Weights

  • Plus le weight est grand, plus le device reçoit de partitions
  • Usage : équilibrer disques hétérogènes (4 TB → weight 100, 8 TB → weight 200)
  • Drainer un disque : weight = 0 + rebalance + remove

Opérations de maintenance Swift

➕ Ajouter un disque à un node

# 1. Préparer le disque côté node
mkfs.xfs -L swift_d3 /dev/sdd
mkdir /srv/node/d3
# Ajouter dans /etc/fstab + mount -a

# 2. Côté builders, ajouter aux 3 rings
swift-ring-builder object.builder add \
  --region 1 --zone 1 \
  --ip 10.10.0.31 --port 6200 \
  --device d3 --weight 100

# 3. Rebalance puis distribuer
# les rings aux nodes (Kolla le fait)
swift-ring-builder object.builder rebalance

kolla-ansible reconfigure -i multinode \
  --tags swift

➖ Retirer un disque proprement

# 1. Drain : weight 0, rebalance
swift-ring-builder object.builder set_weight \
  10.10.0.31/d3 0
swift-ring-builder object.builder rebalance

kolla-ansible reconfigure -i multinode \
  --tags swift

# 2. Attendre que les replicators
# aient déplacé toutes les partitions
# (peut prendre plusieurs heures)
docker exec swift_object_server \
  swift-recon --md5 --all

# 3. Retirer définitivement
swift-ring-builder object.builder remove \
  10.10.0.31/d3
swift-ring-builder object.builder rebalance

⚠️ min_part_hours empêche un nouveau rebalance avant N heures pour éviter les "thrashing" : on attend ou on force avec set_min_part_hours 0 (à utiliser avec prudence).

4. Cinder : architecture

Cinder en quelques mots

🎯 Responsabilités

  • Provisionner des volumes block à la demande (de quelques Go à plusieurs To)
  • Les attacher à des instances Nova comme des disques durs locaux
  • Gérer les snapshots, backups, clones
  • Abstraire des backends hétérogènes : LVM, Ceph, NetApp, EMC, Pure, IBM, NFS… via des drivers
  • Permettre le boot-from-volume (instance dont le root est persistant)
  • Exposer le stockage à Kubernetes (driver CSI Cinder)

🧠 Principes

  • Architecture stateless + scheduler similaire à Nova
  • cinder-volume = agent par backend, parle au stockage physique
  • API REST, intégration Keystone, événements Oslo Messaging (RabbitMQ)
  • Persistance dans une base SQL (MariaDB/Galera)
  • Multi-backend natif : un seul Cinder pilote plusieurs technologies en parallèle

💡 Cinder ne stocke jamais les données lui-même. Il pilote le backend, transmet aux VM via iSCSI/NBD/RBD/FC selon la techno.

6. Types, QoS, snapshots

Qualité de service (QoS) sur les volumes

🎯 Deux niveaux d'application

  • front-end : limite côté hyperviseur (libvirt/QEMU iotune). Universel, fonctionne avec n'importe quel backend
  • back-end : limite côté baie de stockage. Nécessite un driver qui le supporte (NetApp, Pure, Ceph QoS)
  • both : applique les deux

📐 Specs disponibles

  • total_iops_sec
  • read_iops_sec / write_iops_sec
  • total_bytes_sec
  • read_bytes_sec / write_bytes_sec
  • *_max : burst toléré
  • *_burst : durée du burst

💻 Création et association

# 1. Créer une QoS spec
openstack volume qos create gold \
  --consumer front-end \
  --property total_iops_sec=10000 \
  --property total_bytes_sec=524288000

openstack volume qos create bronze \
  --consumer front-end \
  --property total_iops_sec=500 \
  --property total_bytes_sec=52428800

# 2. Associer à un volume type
openstack volume qos associate \
  gold premium

openstack volume qos associate \
  bronze lvm-cold

# 3. Tout volume créé avec le type
# "premium" héritera des limites "gold"
openstack volume create app-db \
  --size 100 --type premium

# Vérifier sur l'hyperviseur
virsh blkdeviotune <instance> vda

6. Types, QoS, snapshots

Snapshots vs Backups

📸 Snapshot

  • Point-in-time du volume au moment T
  • Stocké sur le même backend que le volume parent
  • Copy-on-write sur Ceph/Pure/NetApp : quasi instantané et sans coût
  • Sur LVM : copie réelle, plus lente
  • Permet de cloner un volume
  • Si le backend tombe, snapshot perdu — pas un backup
openstack volume snapshot create \
  --volume app-db db-snap-pre-deploy

openstack volume create \
  --snapshot db-snap-pre-deploy \
  --size 100 app-db-clone

💾 Backup

  • Copie vers un autre stockage (Swift, NFS, Ceph, GCS)
  • Indépendant du backend source : vrai DR
  • Incrémental : seules les diff sont transférées
  • Compressé, optionnellement chiffré
  • Restauration vers un nouveau volume
  • Backup d'un volume en cours d'utilisation = --force
openstack volume backup create \
  --name app-db-2024-12 \
  --incremental app-db

openstack volume backup restore \
  app-db-2024-12 app-db-restored

openstack volume backup list

💡 Stratégie type : snapshots fréquents pour rollbacks rapides (toutes les heures) + backups Swift quotidiens pour DR.

Boot-from-volume : VM avec disque persistant

🎯 Intérêt

  • Le disque système est un volume Cinder, pas un disque éphémère local
  • L'instance survit à la perte d'un compute (rebuild ailleurs)
  • Permet resize sans perte du root
  • Snapshots du disque système
  • Indispensable pour les VM critiques et les pets (vs cattle)

🔧 Variantes

  • Pré-création du volume puis boot dessus
  • Création à la volée du volume depuis une image au moment du boot
  • Boot depuis un snapshot ou un volume existant

💻 Boot from volume — 3 patterns

# 1. Pré-créer le volume depuis une image
openstack volume create root-web-01 \
  --image "Ubuntu 22.04" --size 40 \
  --type ceph-ssd --bootable

openstack server create web-01 \
  --flavor m1.medium \
  --network internal \
  --volume root-web-01

# 2. Créer le volume au boot (one-shot)
openstack server create web-02 \
  --flavor m1.medium --network internal \
  --image "Ubuntu 22.04" \
  --boot-from-volume 40

# 3. Avec --block-device détaillé
openstack server create db-01 \
  --flavor m1.large --network internal \
  --block-device source_type=image,\
uuid=$IMG_ID,destination_type=volume,\
volume_size=80,boot_index=0,\
delete_on_termination=false

Quotas Cinder par projet

📊 Ressources quotables

  • volumes : nombre total de volumes
  • gigabytes : Go totaux sur tous backends
  • snapshots : nombre de snapshots
  • backups : nombre de backups
  • backup_gigabytes
  • volumes_<type> et gigabytes_<type> : quotas par volume type
  • groups : groupes de volumes (consistency)

Les quotas par type permettent par exemple de limiter le SSD coûteux à 200 Go par projet tout en laissant le LVM HDD à 5 To.

💻 Manipulation

# Voir quotas
openstack quota show finance --usage

# Modifier
openstack quota set finance \
  --volumes 50 \
  --gigabytes 5000 \
  --snapshots 100 \
  --backups 20 \
  --backup-gigabytes 2000

# Quotas par volume type
openstack quota set finance \
  --volume-type premium \
  --volumes 10 \
  --gigabytes 200

# Valeurs par défaut
openstack quota show --default

# Lister les volumes d'un projet
openstack volume list \
  --project finance --long

Manila — File Share Service (aparté)

Le troisième pilier du stockage OpenStack, complémentaire de Swift et Cinder

📂 Qu'est-ce que Manila ?

  • Provisionne des partages de fichiers accessibles via NFS, CIFS, CephFS

  • Home dirs partagés pour postes de travail virtualisés
  • Web cluster avec assets partagés (uploads, médias)
  • Backups mutualisés
  • Apps legacy requérant un montage NFS
  • CI/CD : cache partagé entre runners
# Aperçu CLI
openstack share type create default false

openstack share access create web-assets \
  ip 10.0.0.0/24
openstack share show web-assets

Manila est hors du périmètre de ce module, mais c'est l'option à connaître quand un cas d'usage demande "plusieurs VM doivent voir le même filesystem".
