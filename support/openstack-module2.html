1. Nova en profondeur

Nova : le moteur de calcul d'OpenStack

Le service qui transforme une demande d'instance en VM réellement démarrée sur un hyperviseur

🎯 Responsabilités

  • Provisionner et gérer le cycle de vie complet des instances : create, start, stop, reboot, suspend, resize, migrate, delete
  • Planifier le placement des instances sur les nœuds compute selon des filtres et poids
  • Suivre l'inventaire des ressources disponibles (CPU, RAM, disque, PCI) via le service placement
  • Abstraire l'hyperviseur : KVM/QEMU, Hyper-V, ESXi, Ironic (bare-metal), LXC
  • Exposer une API REST (et une API EC2 historique) pour pilotage par CLI / SDK / Horizon

🧠 Principes de conception

  • Shared-nothing : chaque composant est sans état, scalable horizontalement
  • Découplage par message queue : RabbitMQ comme bus interne
  • État centralisé dans une base SQL (MariaDB/Galera)
  • Cells v2 : partitionnement de l'infrastructure pour scaling horizontal
  • Versionné par micro-version d'API (compatibilité ascendante)

💡 Nova est le projet le plus volumineux d'OpenStack — son code est au cœur de la mécanique du Cloud IaaS.

1. Nova en profondeur

Cells v2 — partitionnement pour le scaling

Nova découpe son infrastructure en cellules pour scaler au-delà de quelques centaines de nodes

🧩 Concepts

  • API Cell : nova-api, scheduler, placement, base globale
  • Compute Cells : conductor + DB + RabbitMQ + N compute nodes
  • cell0 : cellule virtuelle qui stocke les instances dont le scheduling a échoué
  • Découplage des bases et des queues : isolation des pannes
  • Scaling jusqu'à 10 000+ nodes par cloud

Depuis Pike, au moins une cell est obligatoire. Kolla-Ansible déploie une seule cell par défaut, suffisante pour la plupart des cas.

1. Nova en profondeur

Le service Placement

Inventaire des ressources allouables — base du scheduling moderne

📦 Modèle de données

  • Resource Provider : entité qui possède des ressources (un hyperviseur, un disque, un PCI device)
  • Resource Class : type de ressource (VCPU, MEMORY_MB, DISK_GB, PCI_DEVICE, VGPU, NET_BW_KILOBIT_PER_SEC)
  • Inventory : quantité totale + step + reserved
  • Allocation : consommation par un consumer (= une instance)
  • Trait : capacité qualitative (HW_CPU_X86_AVX2, COMPUTE_VOLUME_ATTACH, …)
  • Aggregate : groupe logique de providers

🔍 Exemple de requête au scheduler

# Le scheduler demande à Placement :
GET /allocation_candidates?
  resources=VCPU:2,MEMORY_MB:4096,DISK_GB:40
  &required=HW_CPU_X86_AVX2
  &member_of=in:zone-paris

# Réponse : liste de combinaisons
# de providers qui satisfont la demande
{
  "allocation_requests": [
    { "allocations": {
        "compute-01": {
          "resources": {
            "VCPU": 2,
            "MEMORY_MB": 4096,
            "DISK_GB": 40
          }
        }
      }
    }
  ]
}

💡 Placement remplace l'ancien mécanisme où nova-scheduler interrogeait directement la DB compute_nodes. Plus rapide, plus extensible et utilisable par d'autres projets (Neutron pour la BP réseau, Cyborg pour les accélérateurs).

1. Nova en profondeur

nova-scheduler : filters et weighers

🔍 Filters (binaire : oui/non)

Éliminent les hosts qui ne respectent pas une contrainte. Les hosts restants passent à l'étape suivante.

FilterEffet
ComputeFilterHost actif et activé
ComputeCapabilitiesFilterCapacités matérielles compatibles
ImagePropertiesFilterhypervisor_type, architecture alignés
AvailabilityZoneFilterAZ demandée par le flavor
AggregateInstanceExtraSpecsFilterMatch des extra_specs / metadata
ServerGroupAntiAffinityFilterÉvite la co-localisation d'un même group
NUMATopologyFilterCompatibilité NUMA / hugepages
PciPassthroughFilterPrésence des devices PCI demandés

⚖️ Weighers (numérique : score)

Chaque host restant reçoit un score pondéré ; celui avec le meilleur score est élu.

WeigherStratégie
RAMWeigherPlus de RAM libre = score élevé (spread)
DiskWeigherPlus de disque libre = score élevé
CPUWeigherPlus de vCPU libres = score élevé
IoOpsWeigherMoins d'I/O en cours = mieux
MetricsWeigherCustom metrics
BuildFailureWeigherPénalise les hosts qui échouent

Stratégies typiques : spread (RAMWeigher positif) pour la performance, pack (négatif) pour la consolidation.

2. Kolla-Ansible

Kolla-Ansible : rappel et structure

🐳 Principe

  • Chaque service Nova est packagé dans une image Docker officielle (quay.io/openstack.kolla/*)
  • Ansible orchestre le pull, la configuration et le démarrage des conteneurs sur les nodes
  • Les config files sont générés à partir de templates Jinja2 puis montés en bind-mount dans le conteneur
  • Données persistantes dans des volumes Docker nommés (kolla_logs, nova_compute…)

🔑 Fichiers clés

  • /etc/kolla/globals.yml : variables globales
  • /etc/kolla/passwords.yml : tous les mots de passe
  • multinode ou all-in-one : inventaire
  • /etc/kolla/config/nova/* : overrides spécifiques

Structure des overrides Nova

Personnaliser la configuration générée par Kolla sans toucher aux templates

📁 Arborescence /etc/kolla/config/

/etc/kolla/config/
├── global.conf            # Tous services
├── database.conf          # Section [database]
├── messaging.conf         # Section [oslo_messaging_*]
│
└── nova/
    ├── nova.conf          # Tous nodes Nova
    ├── nova-api.conf      # Spécifique nova-api
    ├── nova-compute.conf  # Tous compute
    │
    ├── compute-01/        # Spécifique à un host
    │   └── nova-compute.conf
    │
    └── compute-hyperv-01/
        └── nova-compute.conf

Les overrides sont fusionnés avec les configs générées par Kolla. On n'écrase pas tout, on ajoute/modifie des clés ciblées.

🔧 Exemple : ratios de surcommit

# /etc/kolla/config/nova.conf
# Appliqué à tous les services Nova

[DEFAULT]
cpu_allocation_ratio = 4.0
ram_allocation_ratio = 1.5
disk_allocation_ratio = 1.0

[scheduler]
discover_hosts_in_cells_interval = 300

[filter_scheduler]
enabled_filters = ComputeFilter,
  ComputeCapabilitiesFilter,
  ImagePropertiesFilter,
  ServerGroupAntiAffinityFilter,
  ServerGroupAffinityFilter,
  AggregateInstanceExtraSpecsFilter,
  AvailabilityZoneFilter

[libvirt]
cpu_mode = host-model
virt_type = kvm
disk_cachemodes = network=writeback

2. Kolla-Ansible

Workflow de déploiement / mise à jour

🚀 Premier déploiement

# 1. Bootstrap : prépare les serveurs
# (Docker, sysctl, hugepages, kernel modules)
kolla-ansible bootstrap-servers \
  -i /etc/kolla/multinode

# 2. Pre-checks : valide DNS, NTP, dispo VIP…
kolla-ansible prechecks -i multinode

# 3. Pull les images Docker
kolla-ansible pull -i multinode

# 4. Déploiement complet
kolla-ansible deploy -i multinode

# 5. Post-deploy : génère openrc admin
kolla-ansible post-deploy
source /etc/kolla/admin-openrc.sh

🔄 Reconfiguration ciblée

# Reconfigurer Nova uniquement
kolla-ansible reconfigure -i multinode \
  --tags nova

# Reconfigurer seulement nova-compute
kolla-ansible reconfigure -i multinode \
  --tags nova-compute

# Limiter à un host
kolla-ansible reconfigure -i multinode \
  --tags nova \
  --limit compute-01

# Voir ce qui changerait sans appliquer
kolla-ansible reconfigure -i multinode \
  --tags nova \
  --check --diff

# Mise à jour : nouvelle release
# 1. Maj globals.yml : openstack_release
kolla-ansible pull -i multinode
kolla-ansible upgrade -i multinode

💡 La commande reconfigure est idempotente : Kolla compare l'état souhaité au courant, redémarre uniquement les conteneurs dont la config a changé, et applique les migrations DB nécessaires.

2. Kolla-Ansible

Découverte des compute nodes (cells)

Quand on ajoute un compute, il faut le déclarer dans la cell

⚙️ Mécanisme

  • nova-compute démarre et s'enregistre dans la base nova de la cell
  • Le scheduler ne le voit pas tant qu'il n'a pas été mappé dans nova_api.host_mappings
  • La commande nova-manage cell_v2 discover_hosts fait ce mapping
  • Kolla configure le scheduler pour le faire automatiquement toutes les 5 min : discover_hosts_in_cells_interval=300
  • Mais on peut forcer manuellement pour gagner du temps

🛠️ Commandes pratiques

# Exécuter dans le conteneur nova_api du leader
docker exec -it nova_api bash

# Lister les cells configurées
nova-manage cell_v2 list_cells

# Lister les hosts par cell
nova-manage cell_v2 list_hosts

# Découvrir manuellement
nova-manage cell_v2 discover_hosts \
  --verbose

# Côté client
openstack compute service list
openstack hypervisor list
openstack hypervisor show compute-01

⚠️ Sans discover_hosts, un nouveau compute est invisible au scheduler — toutes les nouvelles instances atterrissent ailleurs. Symptôme typique : "j'ai ajouté un node mais aucune VM ne s'y crée".

3. Flavors & placement

Les flavors : gabarits de VM

🎛️ Composition d'un flavor

  • vCPUs : nombre de cœurs virtuels
  • RAM : mémoire en Mo
  • Disk : disque éphémère root en Go
  • Ephemeral : disque éphémère secondaire
  • Swap : swap en Mo
  • RXTX Factor : multiplicateur de bande passante
  • Extra specs : clés/valeurs pour le scheduling (NUMA, CPU pinning, hyperviseur cible…)
  • is_public : visible par tous ou réservé à des projets

📐 Naming convention typique

m1.tiny, m1.small, m1.medium, c2.xlarge (compute-optimized), r4.large (RAM-optimized), g1.medium (GPU)…

💻 Création via CLI

# Flavor standard
openstack flavor create \
  --vcpus 2 --ram 4096 --disk 40 \
  --ephemeral 0 --swap 0 \
  --rxtx-factor 1.0 --public \
  m1.medium

# Flavor avec extra specs (NUMA + pinning)
openstack flavor create m1.cpu-pinned \
  --vcpus 4 --ram 8192 --disk 40

openstack flavor set m1.cpu-pinned \
  --property hw:cpu_policy=dedicated \
  --property hw:cpu_thread_policy=prefer \
  --property hw:mem_page_size=large \
  --property hw:numa_nodes=1

# Flavor privé pour un projet
openstack flavor create r4.special \
  --vcpus 8 --ram 32768 --disk 100 \
  --private
openstack flavor set r4.special \
  --project finance

3. Flavors & placement

Host Aggregates et Availability Zones

🏷️ Host Aggregates

Groupe logique de compute nodes partageant des propriétés (matériel, licence, fonctionnalité). Invisible aux utilisateurs.

# Créer un aggregate
openstack aggregate create \
  --zone az-paris-1 \
  ssd-nodes

# Tagger l'aggregate
openstack aggregate set \
  --property storage_type=ssd \
  --property gpu=true \
  ssd-nodes

# Ajouter des hosts
openstack aggregate add host \
  ssd-nodes compute-01
openstack aggregate add host \
  ssd-nodes compute-02

# Forcer un flavor à cibler ce groupe
openstack flavor set m1.gpu \
  --property aggregate_instance_extra_specs:gpu=true

🌍 Availability Zones

Vue publique de l'infrastructure : un user peut demander une AZ précise pour ses instances.

  • Une AZ = un aggregate avec la propriété availability_zone=<nom>
  • Un host ne peut être que dans une seule AZ à la fois
  • Usage : isoler des datacenters, des salles, des armoires, des fournisseurs énergie
# Lister les AZ
openstack availability zone list

# Créer une VM dans une AZ donnée
openstack server create \
  --image ubuntu \
  --flavor m1.medium \
  --network internal \
  --availability-zone az-lille-1 \
  web-1

💡 Bonne pratique : utiliser les aggregates pour les caractéristiques techniques (CPU, SSD, GPU) et les AZ pour la résilience géographique.

3. Flavors & placement

Server Groups : affinité et anti-affinité

Placer plusieurs VM ensemble ou les éparpiller pour la HA

📦 Policies disponibles

  • affinity : toutes les VM du groupe sur le même host (perf, latence)
  • anti-affinity : toutes les VM du groupe sur des hosts différents (HA)
  • soft-affinity : best-effort sur même host
  • soft-anti-affinity : best-effort sur hosts différents

Les versions "soft" évitent les NoValidHost : si la contrainte ne peut être satisfaite, l'instance est tout de même créée.

💻 Mise en pratique

# Créer un groupe anti-affinity
openstack server group create \
  --policy anti-affinity \
  web-cluster

# Récupérer l'UUID
GROUP_ID=$(openstack server group list \
  -f value -c ID | head -1)

# Créer 3 instances dans ce groupe
for i in 1 2 3; do
  openstack server create web-$i \
    --image ubuntu --flavor m1.medium \
    --network internal \
    --hint group=$GROUP_ID
done

# Vérifier la répartition
openstack server list --long \
  -c Name -c Host

Migrations : cold, live et evacuate

Cold

Cold Migration

  • L'instance est arrêtée, son disque copié, redémarrée ailleurs
  • Interruption de service mais aucune contrainte technique
  • OK même sans stockage partagé
openstack server migrate \
  web-01 \
  --host compute-03 \
  --wait
Live

Live Migration

  • VM sans interruption visible (downtime ~1s)
  • Nécessite : SSH inter-compute, libvirt TLS/SASL
  • Stockage partagé (NFS, Ceph) ou block-migration
openstack server migrate \
  web-01 --live-migration \
  --host compute-03 \
  --block-migration
Evacuate

Evacuate

  • Rebuild d'une instance dont le compute est HS
  • Possible uniquement si le disque est sur stockage partagé
  • Action manuelle ou via masakari (HA)
# Le compute-01 est en panne
openstack server evacuate \
  web-01 \
  --host compute-02

⚙️ Pré-requis live-migration (Kolla) : enable_nova_ssh=yes, libvirt_enable_sasl=yes, nova_compute_resume_guests_state_on_host_boot=true, ports 16509 (libvirt) et 49152-49215 (TCP) ouverts entre computes.

5. Réseau virtuel

Nova ↔ Neutron : qui fait quoi ?

📞 Interactions

  • À la création d'une instance, nova-compute appelle Neutron pour créer (ou utiliser) un port sur le réseau demandé
  • Neutron renvoie l'IP allouée, le MAC, et les informations VIF (Virtual Interface)
  • nova-compute utilise un VIF driver pour brancher le port au tap/veth approprié de la VM
  • À la suppression, nova-compute demande à Neutron de libérer le port

🔌 VIF types courants (KVM)

  • ovs : connexion via Open vSwitch (br-int)
  • linuxbridge : pont Linux classique
  • vhostuser : DPDK / OVS-DPDK
  • direct : SR-IOV (passthrough)

💻 Manipuler les ports

# Liste des ports d'une VM
openstack port list --server web-01

# Attacher un nouveau NIC à une VM live
openstack server add network \
  web-01 backend-net

# Avec un port pré-créé
openstack port create \
  --network backend-net \
  --fixed-ip subnet=backend-subnet,\
ip-address=10.20.0.50 \
  web-01-eth1

openstack server add port \
  web-01 web-01-eth1

# Détacher
openstack server remove port \
  web-01 web-01-eth1

5. Réseau virtuel

Security Groups : firewall stateful par port

🛡️ Principes

  • Ensemble de règles ingress et egress appliquées sur les ports Neutron
  • Stateful : les réponses sont automatiquement autorisées (conntrack)
  • Plusieurs groupes peuvent être attachés à un port (union des règles)
  • Par défaut : tout sortant autorisé, tout entrant refusé sauf depuis les VM du même groupe

Pas un VPN, pas un router : les security groups filtrent mais ne routent rien. Pour des règles inter-réseaux complexes, utiliser un FWaaS ou un firewall L3.

💻 Exemple : ouvrir HTTP/HTTPS et SSH

openstack security group create web \
  --description "Web stack"

# SSH depuis le réseau d'admin uniquement
openstack security group rule create web \
  --protocol tcp --dst-port 22 \
  --remote-ip 10.0.0.0/24

# HTTP / HTTPS depuis Internet
openstack security group rule create web \
  --protocol tcp --dst-port 80 \
  --remote-ip 0.0.0.0/0
openstack security group rule create web \
  --protocol tcp --dst-port 443 \
  --remote-ip 0.0.0.0/0

# ICMP (ping) depuis tout l'internal
openstack security group rule create web \
  --protocol icmp \
  --remote-ip 10.0.0.0/8

# Appliquer à la VM
openstack server add security group \
  web-01 web

Metadata service et cloud-init

📡 Le metadata service

  • Chaque VM peut interroger http://169.254.169.254/ pour récupérer ses métadonnées : hostname, public keys, user-data, network config
  • Sert à cloud-init, présent dans toutes les images cloud (Ubuntu, Debian, CentOS, Rocky, AlmaLinux…)
  • Le service est rendu par nova-api-metadata (control) via neutron-metadata-agent (routeur)
  • Alternative : config-drive — métadonnées montées en CD-ROM virtuel (utile sans réseau metadata)

Hyper-V — intégration dans OpenStack

🪟 Architecture

  • Le service nova-compute Hyper-V s'installe directement sur Windows Server (avec rôle Hyper-V)
  • Kolla-Ansible n'orchestre pas ce service côté Windows : on le pose hors-Kolla
  • L'hôte Windows publie nova-compute via RabbitMQ et la DB sur la même infrastructure Kolla
  • Distribué via le projet compute-hyperv (anciennement nova-hyperv)

🎯 Quand l'utiliser ?

  • Parc Windows existant à intégrer
  • Charges Windows Server certifiées par Microsoft
  • Intégration Active Directory native
  • Licensing Datacenter favorable

💻 Installation Windows

# Sur le serveur Windows avec rôle Hyper-V
PS> Install-WindowsFeature Hyper-V \
  -IncludeManagementTools -Restart

# Installer le package compute-hyperv
PS> pip install compute-hyperv

[DEFAULT]

VMware ESXi via vCenter

🏗️ Architecture VMwareVCDriver

  • nova-compute parle à vCenter Server (pas directement à ESXi) via l'API SOAP
  • Un seul nova-compute par cluster vCenter — il représente tous les ESXi du cluster
  • Le DRS de vCenter peut continuer à déplacer les VM ; Nova le tolère
  • Stockage : datastores VMFS / vSAN, traduits en disques pour Nova
  • Réseau : NSX ou dvSwitches portés à Neutron via le plugin VMware NSX

🎯 Quand l'utiliser ?

  • Hybridation avec un cluster VMware existant
  • Workloads dépendantes de fonctionnalités VMware (vMotion, DRS, FT)
  • Migration progressive vers KVM

⚙️ Configuration nova-compute

# Sur un compute "passerelle" Linux, hors cluster
# /etc/kolla/config/nova/compute-vmware-01/
#   nova-compute.conf

[DEFAULT]
compute_driver = vmwareapi.VMwareVCDriver

[vmware]
host_ip = vcenter.utopios.local
host_username = administrator@vsphere.local
host_password = SuperSecret!
cluster_name = Production-Cluster
datastore_regex = ^datastore-prod-.*$
api_retry_count = 10
task_poll_interval = 5
use_linked_clone = true
# Réduit le temps de provisionnement

[oslo_concurrency]
lock_path = /var/lib/nova/tmp

⚠️ Compatibilité avec Neutron : par défaut, seul le plugin NSX-T intègre complètement vCenter à Neutron. Sans NSX, les fonctions réseau d'OpenStack (security groups, floating IPs) sont limitées.
