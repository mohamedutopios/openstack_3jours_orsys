1. Le Cloud Computing

Qu'est-ce que le Cloud Computing ?

Définition NIST (National Institute of Standards and Technology)

« Un modèle permettant un accès ubiquitaire, pratique et à la demande, via un réseau, à un pool partagé de ressources informatiques configurables (réseau, serveurs, stockage, applications, services) qui peuvent être rapidement provisionnées et libérées avec un minimum d'effort de gestion ou d'interaction avec le fournisseur. »

Les 5 caractéristiques essentielles du Cloud

📡 Self-service à la demande

L'utilisateur provisionne lui-même ses ressources sans intervention humaine

🌐 Accès réseau étendu

Disponible depuis n'importe quel terminal via le réseau (HTTP/API)

🏊 Mise en commun

Mutualisation des ressources (multi-tenancy) entre clients

⚡ Élasticité rapide

Capacité à monter ou descendre en charge automatiquement

📊 Service mesuré

Facturation à l'usage (pay-as-you-go) et reporting détaillé

🔄 Standardisation

APIs uniformes et automatisables (REST, CLI, SDK)

1. Le Cloud Computing

Apports du Cloud par rapport à l'IT traditionnelle

❌ IT traditionnelle (on-premise)

  • CAPEX lourd : achat de serveurs, licences, datacenter
  • Délais de provisionnement de plusieurs semaines
  • Sur-dimensionnement pour absorber les pics
  • Maintenance matérielle et logicielle à la charge de l'entreprise
  • Évolutivité limitée par les achats budgétaires
  • Compétences silotées (réseau, stockage, virtu, OS)

✅ Cloud Computing

  • OPEX maîtrisé : facturation à l'usage (heure, Go, requête)
  • Provisionnement en quelques minutes via API
  • Élasticité automatique (auto-scaling)
  • Délégation de l'infrastructure au fournisseur
  • Innovation rapide via les services managés
  • Équipes DevOps transverses et automatisation

💡 À retenir

Le Cloud transforme l'infrastructure en un service consommable comme l'électricité. On parle de "computing as a utility". Le défi n'est plus de posséder le matériel mais de l'utiliser efficacement.

1. Le Cloud Computing

Modèles de déploiement

Cloud public, privé, hybride, communautaire — quelle stratégie ?

Public

Cloud Public

Ressources mutualisées chez un fournisseur tiers. AWS, Azure, GCP, OVHcloud.

✓ Avantages

Pas d'investissement initial, scalabilité quasi infinie

✗ Inconvénients

Souveraineté, conformité, coûts à long terme

Privé

Cloud Privé

Infrastructure dédiée à une seule organisation. OpenStack, VMware Cloud.

✓ Avantages

Contrôle total, sécurité, conformité, isolation

✗ Inconvénients

CAPEX, compétences internes, élasticité limitée

Hybride

Cloud Hybride

Mix privé + public avec interopérabilité (VPN, peering, fédération).

✓ Avantages

Flexibilité, cloud bursting, données sensibles isolées

✗ Inconvénients

Complexité, intégration, gouvernance

Communautaire

Cloud Communautaire

Mutualisé entre plusieurs organisations partageant des préoccupations (gov, santé).

✓ Avantages

Mutualisation coûts, conformité sectorielle

✗ Inconvénients

Gouvernance partagée, niche

OpenStack est le standard open source pour bâtir des clouds privés et publics (OVHcloud, Cleura, Infomaniak en sont des exemples publics). Il s'intègre dans des stratégies hybrides via des API compatibles AWS EC2/S3.

2. Le projet OpenStack

OpenStack : définition

OpenStack est une plateforme open source de Cloud Computing qui contrôle de larges pools de ressources compute, stockage et réseau dans un datacenter, le tout administré via un dashboard ou des APIs.

🔓 Open Source

Licence Apache 2.0 — code source publiquement disponible, contributions ouvertes à tous.

🧩 Modulaire

Composé de projets indépendants (Nova, Neutron, Swift…) communiquant par API REST.

🏢 Standard du marché

Soutenu par 500+ entreprises, déployé chez Walmart, CERN, AT&T, China Mobile, Yahoo!.

Cycle de release et versions

📅 Cadence semestrielle

OpenStack publie 2 releases majeures par an, suivant un cycle de 6 mois fortement coordonné entre tous les projets.

Phases du cycle

  • Planning — définition des specs et roadmaps
  • Implementation — milestones M1, M2, M3
  • Feature freeze — stabilisation
  • Release Candidate — tests intégration
  • Release — publication officielle

Convention de nommage

Nom de code alphabétique, généralement lié à la ville hôte du prochain Summit (puis thèmes plus libres).

3. Architecture OpenStack

Nova — Compute Service

Le moteur de calcul d'OpenStack : provisionne et gère les machines virtuelles

🎯 Rôle

  • Provisionner et gérer le cycle de vie des instances (VM, conteneurs LXC, bare-metal via Ironic)
  • Planifier le placement des VM selon des règles (CPU, RAM, anti-affinity)
  • Abstraire les hyperviseurs sous-jacents
  • Exposer une API REST compatible avec Amazon EC2

🏗️ Composants principaux

  • nova-api : entrée REST
  • nova-scheduler : placement des VM
  • nova-conductor : médiateur DB
  • nova-compute : exécution sur les nœuds
  • placement : suivi inventaire ressources

Neutron — Networking

Le SDN d'OpenStack : crée et gère les réseaux virtuels

🎯 Capacités

  • Réseaux L2 isolés (VLAN, VXLAN, GENEVE)
  • Sous-réseaux avec DHCP intégré
  • Routeurs virtuels L3 entre réseaux + Internet
  • Floating IPs (NAT 1:1) pour exposer les VM
  • Security Groups (firewall stateful)
  • Load Balancer (Octavia), VPN, BGP

🔌 Plugins / backends

  • ML2 + drivers : Open vSwitch, Linux Bridge, OVN
  • SDN : Cisco ACI, VMware NSX, Calico

3. Architecture OpenStack

Stockage : Swift, Cinder et Manila

Trois services pour trois paradigmes de stockage

Object

Swift

Stockage objet distribué, accessible via HTTP/REST (API S3-compatible avec Swift3).

Caractéristiques

  • Eventual consistency
  • Réplication 3x par défaut
  • Pas de hiérarchie : containers + objets
  • Scalabilité quasi infinie

Cas d'usage

Backups, photos, vidéos, datasets ML, archives

Block

Cinder

Stockage bloc persistant : volumes attachables aux VM comme des disques.

Caractéristiques

  • Snapshots et clones
  • Quality of Service (QoS)
  • Multi-attach possible
  • 40+ backends (Ceph, NetApp, Pure, LVM…)

Cas d'usage

Disques système, bases de données, datas applicatives

File

Manila

Stockage de fichiers partagé (NFS, CIFS) entre plusieurs instances.

Caractéristiques

  • Protocoles NFS, SMB, CephFS
  • Quotas et règles d'accès
  • Shares accessibles concurrent
  • Optionnel mais utile

Cas d'usage

Partages collaboratifs, codes apps, contenus web

💡 Ceph comme backend unifié : beaucoup d'opérateurs utilisent Ceph (RBD pour Cinder/Glance, RadosGW pour Swift, CephFS pour Manila) — un seul cluster, trois interfaces.

3. Architecture OpenStack

Services transverses : Glance, Keystone, Horizon

Images

🖼️ Glance

Catalogue d'images de VM utilisables pour démarrer les instances.

Formats supportés

  • QCOW2 (KVM, format par défaut)
  • RAW (performance maximale)
  • VHD / VHDX (Hyper-V)
  • VMDK (VMware)
  • AMI / ARI / AKI (compatibilité EC2)
  • ISO (médias d'installation)

Backends de stockage

Local file system, Swift, Ceph RBD, S3, NFS

Identity

🔑 Keystone

Authentification, autorisation et catalogue de services unifié.

Concepts clés

  • User : identité (humain ou service)
  • Project / Tenant : conteneur de ressources
  • Domain : namespace de plus haut niveau
  • Role : permission (admin, member, reader)
  • Token : jeton d'authentification (UUID, Fernet)
  • Catalog : liste des endpoints API

Backends

SQL, LDAP, Active Directory, OIDC/SAML (fédération)

Web UI

📊 Horizon

Dashboard web Django pour piloter OpenStack sans CLI.

Fonctionnalités

  • Gestion VM, volumes, réseaux, images
  • Console VNC/SPICE dans le navigateur
  • Vue admin (hyperviseurs, quotas)
  • Vue projet (mes ressources)
  • Extensible par plugins (panels)

À retenir

Horizon est un consommateur des APIs : tout ce qu'il fait est faisable en CLI / SDK.

3. Architecture OpenStack

Composants techniques sous-jacents

OpenStack repose sur un écosystème de briques tierces matures

🗄️ Base de données

Chaque service stocke son état dans une DB SQL.

  • MariaDB (recommandé) en cluster Galera multi-master
  • PostgreSQL également supporté
  • Une base par service (nova, neutron, glance…)

📨 Message Queue

Bus de communication asynchrone interne.

  • RabbitMQ (par défaut, AMQP)
  • Qpid, ZeroMQ (alternatives)
  • Permet le découplage des composants

⚡ Cache

Accélère les opérations Keystone (tokens) et les autres services.

  • Memcached pour les tokens Fernet
  • Redis dans certaines distributions

🌐 Reverse Proxy / Web

Front HTTP pour les APIs et le dashboard.

  • Apache HTTPD + mod_wsgi (par défaut)
  • Nginx avec uwsgi (déploiements modernes)
  • HAProxy pour la HA des endpoints

🖥️ Hyperviseurs

Couche d'exécution des VM (driver Nova).

  • KVM / QEMU (par défaut, Linux)
  • VMware ESXi (vCenter)
  • Hyper-V, LXC, Xen, Ironic (bare-metal)

🔐 Identity backends

Sources d'identité pour Keystone.

  • SQL (interne, le plus courant)
  • LDAP / Active Directory
  • SAML / OpenID Connect (fédération)

4. Méthodes d'installation

DevStack — pour apprendre et développer

🎯 Pourquoi DevStack ?

  • Installation tout-en-un sur 1 VM ou 1 serveur
  • Idéal pour formation, contribution upstream, démos
  • Active la dernière version en cours de développement (master)
  • Configurable via un fichier local.conf

⚠️ Limites

  • Pas de Haute Disponibilité
  • État perdu au reboot par défaut
  • Performances limitées
  • Jamais à utiliser en production

📋 Prérequis

Ubuntu 22.04 ou Rocky 9 minimum, 8 Go RAM, 60 Go disque, accès root et Internet.

💻 Procédure

# 1. Créer un user dédié
sudo useradd -s /bin/bash -d /opt/stack -m stack
echo "stack ALL=(ALL) NOPASSWD: ALL" | \
  sudo tee /etc/sudoers.d/stack
sudo su - stack

# 2. Cloner le repo
git clone https://opendev.org/openstack/devstack
cd devstack

# 3. Créer local.conf
cat > local.conf <<EOF
[[local|localrc]]
ADMIN_PASSWORD=secret
DATABASE_PASSWORD=\$ADMIN_PASSWORD
RABBIT_PASSWORD=\$ADMIN_PASSWORD
SERVICE_PASSWORD=\$ADMIN_PASSWORD
HOST_IP=10.0.0.10
EOF

# 4. Lancer le déploiement
./stack.sh

# ⏱️ ~30-45 min selon la machine
# Dashboard : http://10.0.0.10/dashboard

Kolla-Ansible — la voie production moderne

🚀 Principe

Tous les services OpenStack sont packagés en conteneurs Docker. Le déploiement est orchestré par Ansible sur l'ensemble des nœuds.

✅ Avantages

  • Isolation : chaque service dans son conteneur
  • Upgrades atomiques : on remplace l'image
  • Rollback facile en cas d'échec
  • HA native : HAProxy + Keepalived + Galera
  • Compatible Docker, Podman
  • Documentation très active

🏗️ Topologie type

  • 3 nœuds control (HA)
  • N nœuds compute
  • 3 nœuds storage (Ceph)
  • 2 nœuds network (optionnel)

💻 Workflow

# 1. Installer Kolla-Ansible
pip install kolla-ansible

# 2. Préparer la config
sudo mkdir -p /etc/kolla
cp -r kolla-ansible/etc_examples/kolla/* \
       /etc/kolla/

# 3. Éditer globals.yml
cat >> /etc/kolla/globals.yml <<EOF
kolla_base_distro: "rocky"
openstack_release: "2024.1"
kolla_internal_vip_address: "10.0.0.250"
network_interface: "eth0"
neutron_external_interface: "eth1"
enable_cinder: "yes"
enable_ceph: "yes"
EOF

# 4. Générer les mots de passe
kolla-genpwd

# 5. Bootstrap des serveurs
kolla-ansible bootstrap-servers \
  -i multinode

# 6. Pre-checks + deploy
kolla-ansible prechecks -i multinode
kolla-ansible deploy    -i multinode
kolla-ansible post-deploy -i multinode
